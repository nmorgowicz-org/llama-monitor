# Gemma4 / MTP Draft Model Integration Plan

Date: 2026-06-07
Status: Active — implementation in progress

## 1. Background

Gemma4 models support Multi-Token Prediction (MTP) via llama.cpp's speculative decoding, using:

- `--spec-type draft-mtp` (or `draft-mtp,ngram-mod`): enables MTP
- `-md <assistant GGUF>`: provides external draft model file (required for Gemma4-style MTP)
- `--spec-draft-n-max N`: max draft tokens per step (recommended: 4 for Gemma4)

Models like Qwen3.6 have internal MTP heads built-in: they use `--spec-type draft-mtp` without `-md`.
Gemma4 uses a separate assistant/draft GGUF file: same family name, smaller size (e.g. ~440 MB vs 16 GB).

Example Gemma4 run:
```
llama-server \
  -m gemma-4-12B-it-qat-UD-Q4_K_XL.gguf \
  --model-draft gemma-4-12B-it-qat-assistant-MTP-Q8_0.gguf \
  --spec-type draft-mtp \
  --spec-draft-n-max 4 \
  ...
```

Goal: fully integrate MTP assistant model support into:
- Spawn wizard (auto-detect, auto-suggest, auto-wire)
- Preset editor (MTP section aware of assistant)
- Models viewer (filter and badge assistant models)
- HF browse (badge assistant files)
- All following the existing mmproj companion pattern.

## 2. MTP Logic (canonical rules)

- `--spec-type draft-mtp`:
  - Enables MTP speculative decoding.
  - For internal-MTP models (e.g. Qwen3.6 with MTP heads), this is sufficient.
- If MTP is NOT internal:
  - You MUST also pass `-md <assistant/draft GGUF>`.
  - The assistant file contains MTP prediction heads used as drafter.
- Combinations (llama.cpp supports):
  - `draft-mtp` alone (internal MTP only)
  - `draft-mtp,ngram-mod` (MTP + n-gram fallback)
  - `draft-model` (external draft model only)
  - Potentially combined: `draft-mtp,draft-model` if both internal + external available.

Implementation rule:
- For models with internal MTP:
  - Offer "Enable MTP" toggle.
  - Use `spec_type = "draft-mtp,ngram-mod"` (default), `n_max = 2`.
- For models that likely need an assistant (Gemma4-style, and any model without internal heads but where assistant candidate exists):
  - Detect companion assistant file.
  - Auto-suggest it in the MTP section.
  - When user enables MTP, if an assistant is present and selected:
    - Use `spec_type = "draft-mtp,ngram-mod"` (or "draft-mtp").
    - Pass `-md` with assistant path.
    - Default `spec_draft_n_max = 4`.
- Allow manual override of n-max (range 1–8, default 4 for assistant-enabled).

## 3. Core Design

- Treat "assistant/draft" GGUF files as a new companion type, parallel to mmproj.
- Hide them from the main model list by default.
- Show them in an "Assistant" section when:
  - Selecting a model with MTP capability
  - Browsing local model directory
  - Browsing HF repo files (badge as "Assistant")

- Matching is generic but conservative:
  - Uses filename similarity (prefix match after normalizing).
  - Not Gemma-only, but tuned to current observed naming patterns.
  - No hard rules about matching quantization.
  - Smaller size relative to main model is a strong hint.

## 4. Assistant Model Detection

A GGUF is considered an assistant/draft candidate if its filename (lowercase):

- Contains: "assistant", "mtp-draft", "draft-model"
- OR matches: "mtp-" prefix + family name (e.g. "mtp-gemma-4-31B-it.gguf")
- OR includes: "mtp_small", "mtp-heads"
- BUT is NOT:
  - A huge file that looks like a main model: > 3 GB or includes tokens like "31B-Q4" at end.
  - Already detected as mmproj.

This heuristic applies across:
- Local model scan (/api/models)
- HF repo file listing (/api/hf/files)
- Spawn wizard directory scan during introspection
- File browser

fn is_draft_assistant_filename(name: &str, size_bytes: u64) -> bool {
    let n = name.to_ascii_lowercase();
    // Obvious assistant indicators
    if n.contains("assistant")
        || n.contains("mtp-draft")
        || n.contains("draft-model")
        || n.contains("mtp_small")
        || n.contains("mtp-heads")
        || n.starts_with("mtp-")

        if size_bytes > 3_000_000_000 {
            return false;
        }
        return true;
    }
    false
}

### 5.9 Frontend: _bestDraftForModel

File: `static/js/features/spawn-wizard.js`

Implement `_bestDraftForModel(mainName, candidates)`:

Rules:
1. Normalize mainName:
   - Remove quant suffix.
   - Remove leading/trailing hyphens.
2. For each candidate:
   - Normalize similarly.
   - Score:
     - Length of common prefix
     - Shared tokens when split by hyphens
     - Prefer candidates:
       - smaller than main model
3. Choose candidate with highest score; if best < 4 chars overlap, no match.

This is generic: works for Gemma4, and future families without being Gemma-specific.

### 5.11 Frontend: buildSpawnPayload — wire draft model

File: `static/js/features/spawn-wizard.js`

Location: `buildSpawnPayload()` (line 7074+).

Changes:
- Current logic (simplified):

  - mtpActive = arch.mtpDepth > 0 && h.mtpEnabled
  - specType = mtpActive ? 'draft-mtp,ngram-mod' : (specTypeSelect value or '')
  - draft_model from dom.draftModelInput.value

We need:
- Determine if assistant is selected:

  - const assistantPath = wizardState.model.selectedDraftPath || (dom.draftModelInput?.value || '').trim();

- Update mtpActive behavior:

  - If arch.mtpDepth > 0 && h.mtpEnabled:
    - mtpActive = true.
    - specType = "draft-mtp,ngram-mod" (or user override).
    - If assistantPath is set:
      - Use it as draft_model.
  - Else if user explicitly selected spec_type includes "draft-mtp" (in advanced):
    - Ensure draft_model uses assistantPath if available.

- spec_draft_n_max:
  - When mtpActive:
    - If assistantPath is set:
      - default: 4
    - else:
      - default: 2

Pseudocode (key parts):
```js
const mtpActive = (arch.mtpDepth > 0 && h.mtpEnabled);
const specTypeUser = dom.specTypeSelect?.value || '';
const mtpSpecBase = 'draft-mtp,ngram-mod';

const hasAssistant = !!wizardState.model.selectedDraftPath;
const draftModelPath = wizardState.model.selectedDraftPath || (dom.draftModelInput?.value || '').trim() || '';

const specType =
  mtpActive
    ? mtpSpecBase
    : (hasAssistant && (specTypeUser?.includes('draft-mtp') || specTypeUser?.includes('draft-model'))
        ? specTypeUser
        : specTypeUser);

// If user manually set advanced spec-type and assistantPath exists,
// we pass both: that's correct behavior for Gemma4.
return {
  ...
  spec_type: specType,
  spec_draft_n_max:
    (mtpActive || hasAssistant)
      ? (h.mtpDraftNMax || (hasAssistant ? 4 : 2))
      : undefined,
  spec_draft_n_min: mtpActive && h.mtpDraftNMin != null ? h.mtpDraftNMin : undefined,
  spec_draft_p_min: mtpActive && h.mtpDraftPMin != null ? h.mtpDraftPMin : undefined,
  draft_model: draftModelPath || '',
  ...
};
```

## 6. Behavior Summary (for future agents)

- For MTP-capable models (via internal heads or assistant):
  - MTP section is shown.
- For models with a local assistant candidate:
  - "Companion assistant" selector appears (dropdown or path input).
  - Selecting or auto-picking it:
    - Sets draft_model in spawn payload.
    - Ensures spec_type includes "draft-mtp".
    - Sets n_max default to 4.
- For HF browse:
  - Assistant files are:
    - Badged as "Assistant".
    - Excluded from auto-selection as main model.
- For models library:
  - Assistant files are:
    - Badged.
    - Optionally filterable, not hidden entirely.

All of this is implemented using the same companion-file pattern currently used for mmproj:
- detected via filename patterns
- hidden from main lists where it would confuse users
- surfaced in the relevant context as a companion option
